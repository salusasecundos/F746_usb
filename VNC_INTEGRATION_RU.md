# VNC/RFB в проекте STM32F746

## Что реализовано

Проект поднимает одноклиентский VNC-сервер на TCP-порту `5900` после получения
IPv4-адреса по DHCP. Сервер работает непосредственно поверх NetX Duo и не
использует BSD sockets, LwIP, FreeRTOS или отдельный framebuffer из проекта
`VNC_TEST_STUFF`.

Поддерживается:

- RFB 3.8, а также совместимое рукопожатие 3.7 и 3.3;
- тип безопасности `None`;
- Raw encoding;
- native RGB565 и запрашиваемые клиентом true-colour форматы 8/16/32 bpp;
- full и incremental `FramebufferUpdateRequest`;
- передача накопленной dirty-области;
- события основной кнопки мыши как `GX_EVENT_PEN_DOWN/DRAG/UP` в GUIX;
- один одновременно подключённый клиент.

Клавиатурные события и clipboard корректно считываются из RFB-потока, но GUI их
пока не использует.

## Правильная точка захвата кадра

Последовательность теперь выглядит так:

1. GUIX полностью дорисовывает скрытый RGB565-буфер.
2. `display_toggle()` назначает его LTDC через
   `HAL_LTDC_SetAddress_NoReload()`.
3. LTDC применяет адрес на vertical blanking.
4. Именно после этого `App_VNC_Frame_Publish()` копирует завершённый кадр в
   отдельный RFB snapshot.
5. GUIX синхронизирует второй display buffer по dirty-области и начинает
   рисовать следующий кадр.

Нельзя просто передать VNC-потоку указатель на front buffer: после следующего
переключения этот же массив станет back buffer, и GUIX начнёт менять его во
время медленной TCP-передачи. Результатом будут разрывы кадра и случайные смеси
двух поколений изображения.

Для устранения гонки используются три неизменяемых RFB snapshot-буфера в
кэшируемой SDRAM. Пока сеть читает один snapshot, GUI может последовательно
публиковать новые кадры через два остальных. GUI-поток никогда не ждёт TCP.

Размер памяти:

- два LTDC/GUIX framebuffer: `2 * 480 * 272 * 2 = 522240` байт;
- три RFB snapshot: `3 * 480 * 272 * 2 = 783360` байт;
- суммарно для кадров: `1305600` байт внешней SDRAM.

## Файлы интеграции

- `Core/Src/app_guix.c` — публикация после VBlank и ввод мыши в GUIX;
- `Core/Src/app_vnc_frame.c` — тройной immutable snapshot и dirty-область;
- `Core/Inc/app_vnc_frame.h` — интерфейс владения snapshot;
- `NetXDuo/App/app_vnc_server.c` — RFB state machine и передача через NetX Duo;
- `NetXDuo/App/app_vnc_server.h` — порт и запуск сервера;
- `NetXDuo/App/app_netxduo.c` — запуск VNC после инициализации TCP/IP.

## Подключение

1. Подключить Ethernet и дождаться DHCP-адреса на вкладке Network.
2. В VNC Viewer открыть `<IPv4-адрес-платы>:5900`.
3. Если viewer предлагает выбор качества/encoding, Raw поддерживается всегда;
   преобразование RGB565 в выбранный true-colour формат выполняет плата.

Порт можно переопределить через `APP_VNC_SERVER_PORT` до включения
`app_vnc_server.h`.

## Безопасность

Сейчас используется RFB security type `None`: пароль и шифрование отсутствуют.
Такой режим подходит только для доверенной изолированной LAN. Для подключения
через недоверенную сеть нужен VPN/туннель либо отдельная реализация VNC
authentication/TLS.
